## 2. 核心业务流程 (Business Flow)

*(提示：使用有序列表，清晰描述数据在前后端的流转过程)*

1. 用户在前端页面点击“忘记密码”。
2. 前端调用 `/api/v1/auth/send-code` 接口发送短信。
3. 后端生成 6 位随机验证码，存入 Redis（设置 5 分钟过期），并调用短信服务发送。
4. 用户输入验证码和新密码，前端调用 `/api/v1/auth/reset-password` 接口。
5. 后端校验 Redis 中的验证码，成功则加密更新数据库中的用户密码，并返回成功状态。

## 3. 数据与字段要求 (Data Requirements)

*(提示：在这里把关键字段的中英文定死，防止 AI 自由发挥导致前后端对不上)*

- **User Entity (用户表新增或依赖字段)**:
    - `phone` (String): 手机号，必填，需正则校验格式。
    - `password` (String): 登录密码，必须使用 BCrypt 加密存储，绝不能明文。
- **DTO (前后端交互对象)**:
    - `ResetPasswordDTO`: 包含 `phone`, `code`, `newPassword`。

## 4. UI 与交互约束 (UI/UX Constraints)

- **页面路由**: `/forgot-password`
- **组件要求**: 验证码输入框必须有 60 秒倒计时防刷机制。
- **错误提示**: 验证码错误时，前端需拦截并使用 Toast 提示“验证码错误或已过期 (Invalid or expired code)”。

## 5. 验收标准 (Acceptance Criteria)

*(提示：这是 AI 写完代码后进行自我检查的红线)*

- [ ] 密码**必须**加密入库。
- [ ] 验证码的 Redis 缓存**必须**有且仅有 5 分钟的 TTL。
- [ ] 接口响应**必须**符合全局的 `{code, message, data}` 统一 JSON 格式。
- [ ] 前端提交请求时，按钮必须有 `loading` 状态防止重复点击。
